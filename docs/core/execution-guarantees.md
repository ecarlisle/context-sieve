# Execution Guarantees

This document defines what the execution layer guarantees and, equally importantly, what it does NOT guarantee.

---

## Invariants (Immutable Guarantees)

### 1. Pipeline Order Is Fixed

Stage order is determined at construction time and never changes during execution.

```
collect → measure → budget → summarize → prune → dedupe → compress → retrieve → forward
```

- Stages cannot be reordered dynamically.
- Plugin stages insert at fixed positions (before/after a named built-in stage).
- No stage can skip another stage.
- No stage can execute more than once per run.

**Violation:** If you observe stages running in a different order, the pipeline was constructed incorrectly. This is a programming error, not a runtime condition.

### 2. Snapshots Are Immutable

Once written, a `RunSnapshot` is never modified.

- Snapshots are write-once JSON files on disk.
- No stage, plugin, or analysis function can alter a snapshot after creation.
- Annotations are stored separately in SQLite and overlaid at replay time.
- Deletion is the only destructive operation (and it removes the entire snapshot).

**Rationale:** Immutability guarantees that replay, analysis, and comparison always operate on the exact same artifact. Without immutability, you cannot trust that a replay reflects what actually happened.

### 3. Execution Is Deterministic Per Run

Within a single execution (`pipeline.run(ctx)`), given the same inputs:
- Stage order is deterministic.
- Stage results are determined by stage logic (some stages may use random seeds).
- The trace (`StageResult[]`) is fixed once produced.
- The snapshot ID is unique per execution (timestamp + random suffix).

**Note:** Cross-execution determinism is NOT guaranteed. Two executions with identical inputs may produce different pruning decisions if the prune stage uses randomization. This is by design — it prevents systematic bias.

### 4. Providers Are Transport-Only

A provider adapter:
- Receives an optimized `ChatCompletionRequest`
- Sends it to the external API
- Returns a `ChatCompletionResponse`

A provider adapter CANNOT:
- Modify the pipeline
- Skip stages
- Read or write snapshots
- Access other providers' data
- Alter routing decisions
- Change the execution order

**Violation:** If a provider adapter does any of the above, it violates the architecture boundary. Adapters are isolated and swappable by design.

### 5. Stages May Not Reorder Dynamically

No stage can change the execution order of subsequent stages. The pipeline iterates through stages sequentially:

```typescript
for (const stage of stages) {
  const result = await stage.run(ctx)
  trace.push(result)
  if (result.status === 'error') break
}
```

- A stage cannot skip the next stage.
- A stage cannot insert a new stage mid-execution.
- A stage cannot re-run a previous stage.

### 6. Decisions Never Block Execution

Decision-related metadata (advisory scores, summary confidence, prune reasons) is informational. It is recorded in the trace but never used to block or abort execution.

- A pruning advisory does not halt the pipeline.
- A low-confidence summary does not prevent forwarding.
- A stage error returns an error result but does not prevent subsequent stages from running (the forward stage may still attempt execution).

**Exception:** The budget stage can abort execution if a hard budget limit is exceeded (configurable). This is explicit and logged.

## What CAN Mutate

| Component | Can Mutate | Details |
|-----------|-----------|---------|
| Pipeline stages | `PipelineContext.state` | Key-value store for stage-to-stage communication within a single run |
| MetricsCollector | Internal counters | Request count, token estimates, latency |
| Storage (SQLite) | Request trace table | Append-only insert of request metadata |
| SnapshotStore | Snapshots directory | Write-once creation of snapshot files |
| Log output | stdout/stderr | Verbose mode prints stage results |
| ReplayStore | Replay artifacts directory | Write-once creation of replay files |

## What CANNOT Mutate

| Component | Cannot Mutate | Rationale |
|-----------|--------------|-----------|
| Any stage | RunSnapshot (after creation) | Immutability guarantee |
| Any stage | Replay artifacts | Generated by replay, not execution |
| Any plugin | Pipeline stage order | Fixed at construction |
| Any plugin | Provider routing decision | Resolved by registry, not by plugins |
| Any adapter | PipelineContext.state | Adapters have no access to pipeline state |
| Any adapter | Snapshots, metrics, storage | Adapters are transport-only |
| Analysis tools | Snapshots, replay artifacts | Read-only by design |

## Execution Lifecycle

```
1. RECEIVE request
        │
2. CREATE PipelineContext
   │  ├─ request (from client)
   │  ├─ config (from defaultConfig)
   │  ├─ metrics (zeroed counters)
   │  ├─ state (empty Map)
   │  └─ response (undefined)
   │
3. RUN pipeline
   │  for each stage:
   │    ├─ stage.run(ctx) → StageResult
   │    ├─ push result to trace[]
   │    └─ break on error (optional)
   │
4. CAPTURE snapshot
   │  ├─ captureRunSnapshot(ctx, trace) → RunSnapshot
   │  └─ snapshotStore.saveSnapshot(snapshot) → disk (write-once)
   │
5. RETURN response + trace
   │
6. RECORD metrics
   │  ├─ metrics.record({...})
   │  └─ storage.insertRequestTrace({...})
   │
7. (Optional) GENERATE replay
      └─ ReplayStore stores replay artifacts
```

---

## What This Means for Correctness

- **You can trust snapshots.** They are immutable records of exactly what happened.
- **You can trust replay.** It projects the snapshot without re-execution.
- **You can trust stage ordering.** It is fixed and observable in the trace.
- **You cannot trust cross-run equivalence.** The same input may produce different outputs across separate executions.
- **You cannot trust provider behavior.** The provider response is external and not under context-sieve's control.

## Common Failure Modes

**"The replay shows different stage order than I expected."**
The pipeline was constructed with a different stage array. Check Pipeline construction in `src/index.ts`.

**"I modified a snapshot file and now replay is broken."**
Snapshots are immutable. If you edit a snapshot file manually, you corrupt the artifact. Delete it and re-run.

**"Two identical requests produced different snapshots."**
This is expected. Stage logic (especially pruning) may use randomization. Compare the snapshots with `context-sieve diff` to see exactly what differed.

**"A plugin changed the stage order."**
This cannot happen. Plugin stages insert at predetermined positions and cannot reorder built-in stages. If you observe order changes, the pipeline construction code is wrong.
